# Knowledge Capture Skill

**대화를 Notion 문서로 변환** - 회의, 토론, 학습에서 나온 지식을 체계적으로 정리하고 저장합니다.

---

## 🎯 이 스킬이 뭔가요?

진행 중인 **대화와 토론에서 나오는 지식**을 자동으로 **구조화된 문서**로 변환하여 Notion에 저장합니다.

그저 채팅 기록에 남겨두는 대신, 팀 전체가 쉽게 찾아보고 참고할 수 있는 **체계적인 지식 베이스**로 만듭니다.

---

## 🔄 어떻게 작동하나요?

### 6단계 프로세스

```text
1. 콘텐츠 추출 → 2. 타입 결정 → 3. 구조화 →
4. 목적지 선택 → 5. 페이지 생성 → 6. 검색 가능하게 만들기
```

#### 1단계: 대화에서 콘텐츠 추출

```
현재 대화에서 가치 있는 정보:

💡 핵심 개념
  - 주요 개념과 정의
  - 중요한 개념들

📌 의사결정 사항
  - 결정 내용
  - 결정의 근거
  - 고려한 대안들

📚 How-To 정보
  - 절차와 단계
  - 팁과 주의사항
  - 자주 하는 실수

🎓 학습과 인사이트
  - 배운 점
  - 모범 사례
  - 나중에 참고할 내용

❓ 자주 하는 질문
  - Q&A 쌍
  - 명확화가 필요한 부분

📋 회의 요약
  - 주요 논의 사항
  - 결론
  - 액션 아이템
```

#### 2단계: 콘텐츠 타입 결정

```
어떤 종류의 문서인지 분류:

🔹 개념 (Concept)
   무엇인가를 설명
   예: "마이크로서비스 아키텍처란"

🔹 How-To 가이드
   어떻게 하는지 설명
   예: "프로덕션에 배포하는 방법"

🔹 의사결정 기록
   왜 결정했는지 설명
   예: "왜 TypeScript를 선택했는가"

🔹 FAQ
   자주 하는 질문과 답변
   예: "API 인증 방식 FAQ"

🔹 학습 기록
   무엇을 배웠는지 설명
   예: "데이터베이스 마이그레이션 회고"

🔹 참고 문서
   참고할 정보 정리
   예: "팀 온보딩 가이드"
```

#### 3단계: 콘텐츠 구조화

```
타입별 구조:

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📝 개념 (Concept)
├── 개요 (1-2문장)
├── 정의 (명확한 설명)
├── 특징 (핵심 특성)
├── 예시 (구체적 예)
├── 사용 사례 (언제 쓰나)
└── 관련 내용 (링크)

📋 How-To 가이드
├── 개요 (이 가이드의 목적)
├── 사전 준비 (필요한 것)
├── 단계별 절차 (1, 2, 3...)
├── 검증 (완료했는지 확인)
├── 문제 해결 (자주하는 실수)
└── 관련 가이드 (링크)

⚖️ 의사결정 기록
├── 배경 (왜 결정해야 했나)
├── 의사결정 (무엇을 결정했나)
├── 근거 (왜 이렇게 결정했나)
├── 검토한 대안 (다른 선택지)
├── 영향 (결과와 파급효과)
└── 구현 (어떻게 실행)

❓ FAQ
├── 짧은 답변
├── 상세 설명
├── 예시
├── 언제 써야 하나
└── 관련 질문 (링크)

🎓 학습 기록
├── 무엇이 일어났나
├── 잘된 점
├── 잘못된 점
├── 근본 원인
├── 배운 점
└── 개선 액션

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

#### 4단계: 저장 목적지 선택

```
어디에 저장할지 결정:

🏠 일반 위키 (General Wiki)
   팀 전체가 참고하는 지식 베이스
   예: "팀 모범 사례", "시스템 아키텍처"

📂 프로젝트 위키
   특정 프로젝트에만 관련된 문서
   예: "모바일 앱 개발 가이드"

🗄️ 문서 데이터베이스
   카테고리, 태그, 상태로 정렬된 문서
   예: 기술 문서, 설계 리뷰

❓ FAQ 데이터베이스
   구조화된 Q&A
   예: "API FAQ", "배포 관련 FAQ"

📋 의사결정 로그
   아키텍처/제품 의사결정 기록
   예: "기술 선택 결정 로그"

👥 팀 위키
   팀 특화 지식
   예: "엔지니어링 팀 온보딩"
```

#### 5단계: Notion에 페이지 생성

**도구**: `Notion:notion-create-pages`

```
Notion에 저장:

✅ 적절한 제목 설정
✅ 구조화된 콘텐츠로 작성
✅ 데이터베이스라면 속성 설정
   - 타입, 카테고리, 태그
   - 생성 날짜, 작성자
   - 상태 (미발행/발행됨)
✅ 관련 페이지 링크
✅ 메타데이터 추가
```

#### 6단계: 검색 가능하게 만들기

**도구**: `Notion:notion-update-page`

```
다른 사람들이 찾을 수 있게:

1️⃣ 허브/인덱스 페이지 업데이트
   ├─ Wiki 목차 페이지에 링크 추가
   ├─ 관련 프로젝트 페이지에서 링크
   └─ 카테고리/주제 페이지에서 링크

2️⃣ 데이터베이스에 있다면
   ├─ 태그/카테고리 설정
   ├─ 상태를 "발행됨"으로 설정
   └─ 관련 뷰에 추가

3️⃣ 부모 페이지 업데이트 (선택)
   ├─ 프로젝트라면 "문서" 섹션 업데이트
   └─ 팀 위키라면 팀 홈페이지에서 링크

⚡ 핵심: 페이지가 고아(orphaned)가 되지 않도록
      워크스페이스 네비게이션 구조에 연결되어야 함
```

## 📊 콘텐츠 타입별 구조

| 타입 | 구조 | 용도 |
| --- | --- | --- |
| **개념** | 개요 → 정의 → 특징 → 예 → 사용 사례 | 무엇을 배우기 |
| **How-To** | 개요 → 사전 준비 → 단계 → 검증 → 문제해결 | 어떻게 하기 |
| **의사결정** | 배경 → 결정 → 근거 → 대안 → 영향 | 왜 결정했나 |
| **FAQ** | 짧은 답 → 상세설명 → 예 → 언제 쓸지 | 자주하는 질문 |
| **학습** | 무엇 → 잘된점 → 실패 → 원인 → 배운점 | 경험에서 배우기 |

## 💡 실제 사용 예시

### 예시 1: 대화에서 FAQ 만들기

```
상황:
팀 채팅에서 API 인증 방식에 대해 여러 질문이 나옴.
같은 내용을 계속 반복 설명하고 있음.

프로세스:
1️⃣ 추출: 대화에서 Q&A 쌍 모으기
   Q: "API 토큰은 어디서 생성하나요?"
   A: "대시보드의 API 키 섹션에서..."

2️⃣ 타입: "FAQ"로 분류

3️⃣ 구조화:
   - 짧은 답변 (한 문장)
   - 상세 설명 (스크린샷 포함)
   - 예시 코드
   - 언제 쓰는지
   - 관련 FAQ 링크

4️⃣ 목적지: "API FAQ" 데이터베이스

5️⃣ 생성: FAQ 데이터베이스에 10개 항목 추가

6️⃣ 발견성:
   - Wiki 목차의 "API" 섹션에 링크
   - 온보딩 가이드에서 "API FAQ 참고" 언급
   - 각 FAQ는 관련 항목끼리 링크

결과:
└── API FAQ (10개 항목 구조화)
    ├─ 토큰 생성
    ├─ 권한 설정
    ├─ 오류 처리
    └─ ...
```

### 예시 2: 토론에서 의사결정 기록 만들기

```
상황:
팀 회의에서 데이터베이스 선택을 결정함.
PostgreSQL vs MongoDB 중 PostgreSQL 선택.

프로세스:
1️⃣ 추출: 논의 내용 정리
   - 각 옵션의 장단점
   - 최종 결정
   - 결정 이유

2️⃣ 타입: "의사결정 기록"

3️⃣ 구조화:
   - 배경: 왜 결정 필요했나
   - 의사결정: PostgreSQL 선택
   - 근거: 복잡한 쿼리 지원 필요
   - 검토 대안: MongoDB, MySQL
   - 영향: 스키마 설계 방식, 마이그레이션 계획
   - 구현: 마이그레이션 일정

4️⃣ 목적지: "의사결정 로그" 데이터베이스

5️⃣ 생성: 의사결정 기록 페이지 생성

6️⃣ 발견성:
   - 기술 결정 페이지에서 링크
   - 아키텍처 페이지의 "데이터베이스" 섹션에서 참고
   - 마이그레이션 계획에서 "결정 배경" 링크

결과:
└── 의사결정: 데이터베이스 선택
    ├─ 배경 및 요구사항
    ├─ 검토한 옵션 비교표
    ├─ 최종 선택 근거
    └─ 구현 타임라인
```

### 예시 3: 회의에서 How-To 만들기

```
상황:
팀 회의에서 프로덕션 배포 절차를 상세히 논의.
새로운 팀원들을 위해 가이드 필요.

프로세스:
1️⃣ 추출: 배포 단계, 주의사항, 문제해결
   - 사전 체크리스트
   - 배포 명령어
   - 검증 방법
   - 실패 시 대응

2️⃣ 타입: "How-To 가이드"

3️⃣ 구조화:
   - 개요: 이 가이드의 목표
   - 사전 준비: 필요한 권한, 도구
   - 단계별:
     1. 준비 단계
     2. 배포
     3. 검증
   - 검증: 배포 성공 확인 방법
   - 문제해결: 자주하는 오류

4️⃣ 목적지: "배포 가이드" 일반 위키

5️⃣ 생성: 구조화된 가이드 페이지

6️⃣ 발견성:
   - Wiki 목차의 "배포" 섹션에 링크
   - 온보딩 체크리스트에서 "배포 가이드 읽기" 항목
   - DevOps 대시보드에서 링크

결과:
└── How-To: 프로덕션 배포
    ├─ 체크리스트
    ├─ 단계별 명령어
    ├─ 검증 방법
    ├─ 일반적인 실수
    └─ 빠른 참고표
```

## 🚀 시작하기

### 사용자 입장

```
"방금 대화에서 나온 [내용]을
Notion에 [형식]으로 저장해 줄래?
[저장 위치] 쪽에 저장해."
```

**예시:**
```
"우리가 방금 토론한 API 인증 방식을
FAQ 형식으로 정리해 줄래?
API FAQ 데이터베이스에 저장해."

"팀 회의에서 PostgreSQL 선택 결정을
의사결정 기록으로 저장해 줄래?
기술 결정 로그에 저장해."

"새 팀원 온보딩 절차를
How-To 가이드로 정리해 줄래?
팀 위키의 온보딩 섹션에 저장해."
```

### 기대할 수 있는 것

1. **구조화된 문서**: 타입별 적절한 형식으로 정리
2. **쉬운 발견성**: 링크와 메타데이터로 쉽게 찾음
3. **일관성**: 같은 타입은 같은 형식으로
4. **완전한 맥락**: 왜, 무엇을, 어떻게를 모두 포함
5. **팀 공유**: 모든 팀원이 접근 가능한 지식 베이스
6. **미래 참고**: 언제든 다시 찾아볼 수 있는 형태

## 📚 스킬의 핵심 철학

> **"대화는 순간, 지식은 영원"**

- 채팅은 **일시적**: 시간이 지나면 스크롤되어 사라짐
- 문서는 **영구적**: 언제든 참고 가능
- 구조는 **검색성**: 체계적 형식으로 쉽게 찾음
- 링크는 **연결성**: 관련 지식들이 상호 연결

**결과**: 팀의 집단 지혜를 축적하고 재사용 가능한 형태로 보존

## ✅ 모범 사례

1. **빠르게 캡처**: 대화 직후 기억이 생생할 때 기록
2. **타입 명확히**: 어떤 종류의 문서인지 먼저 결정
3. **템플릿 사용**: 일관된 형식으로 유지
4. **예시 포함**: 추상적 설명보다 구체적 예
5. **광범위 링크**: 관련된 모든 문서와 연결
6. **쓰기 검색용**: 제목과 태그를 검색하기 쉽게
7. **메타데이터 추가**: 날짜, 작성자, 상태 명시
8. **정기적 검토**: 오래된 문서 업데이트 여부 확인
9. **피드백 요청**: 문서가 도움이 되는지 확인
10. **중복 제거**: 저장하기 전에 기존 문서 검색

## ❓ FAQ

**Q: 저장할 위치를 모르면 어떻게 하나요?**

A: 기본값 사용:
- 불확실하면 **일반 위키에 저장** (나중에 이동 가능)
- 사용자에게 선호 위치 물어보기
- 각 위치의 특징 설명해주고 선택하게 하기

**Q: 이미 같은 내용이 있으면?**

A: 통합하기:
- 먼저 기존 문서 검색
- 기존 문서 업데이트 (새 내용 추가)
- 혹은 새 버전 생성 (상세하면)
- 두 문서가 서로 링크되도록 설정

**Q: 너무 짧은 내용인데 문서로 만들까요?**

A: 기준:
- 2문단 이상 → 문서 생성
- 1문단 이하 → FAQ나 기존 문서에 추가
- 혹은 관련된 내용들 모아서 문서 생성

**Q: 대화가 비공식적/캐주얼한데?**

A: 정제하기:
- 핵심 정보는 유지
- 언어를 전문적으로 정리
- 예시와 맥락은 보강
- 느낌(톤)은 친근하게 유지

**Q: 언제 새로 만들고 언제 업데이트하나요?**

A: 기준:
- **새 문서**: 실질적 내용 (2문단+), 여러 번 참고될 예정, 독립적 검색 필요
- **기존 업데이트**: 같은 주제 확장, 정보 수정, 새 예시 추가
- **버전 관리**: 큰 변경은 변경 이력 섹션 추가

**Q: 개인적인 내용도 저장해도 되나요?**

A: 맥락에 따라:
- **팀 전체 관련**: 공개 위키에 저장
- **팀 내부만**: 팀 위키에 저장
- **개인 노트**: 개인 노트북에 저장
- **민감한 내용**: 권한 제한 설정

**Q: 진행 중인 프로젝트의 내용도 저장할까요?**

A: 프로젝트 단계별:
- **초기 단계**: 프로젝트 페이지에만 기록
- **확정된 결정**: 의사결정 로그에 저장
- **진행 완료**: 완료 후 회고/학습 기록
- **Best Practice**: 유용하면 일반 위키에도 저장
