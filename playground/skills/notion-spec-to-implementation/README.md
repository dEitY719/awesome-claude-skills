# Spec to Implementation Skill

**사양을 구현 계획으로 변환** - 제품 사양서를 구체적인 작업과 타임라인으로 전환하여 개발을 체계적으로 진행합니다.

---

## 🎯 이 스킬이 뭔가요?

제품/기술 **사양서를 읽고 분석**한 후, 이를 **실행 가능한 구현 계획**으로 변환합니다.

단순히 요구사항을 나열하는 것이 아니라, 담당자가 실제로 구현할 수 있도록 **세부 작업**, **우선순위**, **의존성**, **성공 기준**을 명확하게 정의합니다.

---

## 🔄 어떻게 작동하나요?

### 8단계 프로세스

```text
1. 사양서 찾기 → 2. 사양 분석 → 3. 요구사항 추출 →
4. 계획 수립 → 5. 작업DB 찾기 → 6. 작업 생성 →
7. 구현 시작 → 8. 진행 추적
```

#### 1단계: 사양서 찾기

**도구**: `Notion:notion-search`

```
사양서 검색:
- 사양 이름이나 주제로 검색
- 필터 적용 (날짜, teamspace 등)
- 사양서 제목이나 키워드 매칭
- 찾지 못하면 사용자에게 URL/ID 요청

검색 예시:
- "사용자 인증 사양"
- "결제 통합 스펙"
- "모바일 앱 리디자인 PRD"
```

#### 2단계: 사양서 분석

**도구**: `Notion:notion-fetch`

```
사양서 읽고 분석:

📋 기능 요구사항
  - 사용자 스토리
  - 기능 설명
  - 워크플로우
  - 데이터 요구사항
  - 통합 포인트

⚙️ 비기능 요구사항
  - 성능 목표
  - 보안 요구사항
  - 확장성 필요성
  - 가용성
  - 규정 준수

✅ 수용 기준
  - 테스트 가능한 조건
  - 사용자 검증 포인트
  - 성능 벤치마크
  - 완료 정의
```

#### 3단계: 요구사항 추출

```
사양에서 추출할 항목:

✓ 기능 요구사항: 무엇을 만들 것인가?
✓ 수용 기준: 어떻게 알 수 있나 잘했는가?
✓ 의존성: 먼저 완료되어야 할 것은?
✓ 막힘 요소: 어떤 리스크가 있는가?
✓ 제약 조건: 어떤 제한이 있는가?
```

#### 4단계: 구현 계획 수립

**도구**: `Notion:notion-create-pages`

```
구현 계획 작성:

📌 개요 (사양서 링크 포함)

📋 요구사항 요약
  - 기능 목록
  - 수용 기준

🔧 기술 접근법
  - 구현 전략
  - 기술 선택
  - 아키텍처

📅 구현 단계
  단계 1: [마일스톤]
  - 목표: ...
  - 작업: [ ] [ ] [ ]
  - 예상 노력: ...

  단계 2: [마일스톤]
  ...

⚠️ 의존성 & 리스크
  - 선행 작업
  - 리스크 및 완화 방안

📊 타임라인

✨ 성공 기준
```

#### 5단계: 작업 DB 찾기

**도구**: `Notion:notion-search`

```
작업 데이터베이스 검색:
- "작업", "작업 관리", "Task" 등으로 검색
- 엔지니어링/프로젝트 작업 추적 시스템 찾기
- 찾지 못하면 사용자에게 위치 요청

DB 스키마 파악:
- 속성명, 타입, 옵션 확인
- 필수 속성 파악
- 올바른 data_source_id 식별
```

#### 6단계: 구현 작업 생성

**도구**: `Notion:notion-create-pages` (작업 DB에)

```
각 작업마다:

✅ 작업명
  - 명확한 설명
  - 1-2일 내 완료 가능한 크기

📊 속성 설정
  - 상태: To Do
  - 우선순위: P0/P1/P2
  - 관련 작업: 사양서 및 계획 링크

📝 작업 상세
  - 구현할 것
  - 수용 기준
  - 기술 노트
  - 의존성
```

#### 7단계: 구현 시작

```
첫 작업 시작:

1️⃣ 작업 상태를 "진행 중"으로 변경
2️⃣ 초기 진행 노트 추가
3️⃣ 접근 방식과 결정사항 기록
4️⃣ 관련 리소스 링크
```

#### 8단계: 진행 추적

```
정기적 업데이트:

📅 일일 업데이트 (활발한 작업)
  - 완료된 항목
  - 현재 진행 중인 작업
  - 막힘 요소

🎯 마일스톤 업데이트 (주요 진행)
  - 계획 체크박스 업데이트
  - 마일스톤 요약
  - 타임라인 조정

✅ 상태 변경 (작업 전환)
  - 속성 업데이트 (진행 중 → 검토 중 → 완료)
  - 완료 노트 추가
  - 결과물 링크 (PR, 설계, 문서)
```

## 💡 실제 사용 예시

### 예시 1: API 기능 구현

```
요청:
"API 스펙을 구현 계획으로 변환해 줄래?
사용자 인증 API 엔드포인트 3개가 필요해."

프로세스:
1️⃣ 검색: Notion에서 "사용자 인증 API 스펙" 찾기
2️⃣ 분석: 엔드포인트, 요구사항, 보안 요구사항 분석
3️⃣ 계획: 3단계 구현 계획 수립
   - 단계 1: 기본 인증 로직
   - 단계 2: 토큰 관리
   - 단계 3: 테스트 및 배포
4️⃣ 작업: 각 단계별 작업 생성
5️⃣ 시작: 첫 번째 작업 "인증 스키마 설계" 시작

결과:
└── 구현 계획: 사용자 인증 API
    ├── 3단계 계획
    ├── 15개 구체적 작업
    ├── 우선순위 명시
    └── 의존성 표시
```

### 예시 2: UI 컴포넌트 구현

```
요청:
"대시보드 UI 컴포넌트 사양을 구현 작업으로 분해해 줄래?"

프로세스:
1️⃣ 검색: "대시보드 UI 설계 사양" 찾기
2️⃣ 분석:
   - 필요한 컴포넌트들
   - 반응형 요구사항
   - 접근성 요구사항
3️⃣ 계획:
   - 단계 1: 기본 컴포넌트 구조
   - 단계 2: 스타일링 및 반응형
   - 단계 3: 데이터 통합
4️⃣ 작업:
   - Button, Card, Modal 등 각 컴포넌트별
   - 스토리북 문서
   - 접근성 테스트
5️⃣ 추적: 각 컴포넌트 완료도 추적

결과:
└── 구현 계획: 대시보드 UI
    ├── 컴포넌트별 작업
    ├── 디자인 검토 마일스톤
    ├── 통합 테스트
    └── 배포 준비
```

요청:

프로세스:

4. **진행 추적**: Notion에서 실시간 진행 상황 확인
5. **변경 관리**: 사양 변경의 영향 파악

## 📚 스킬의 핵심 철학

> **"사양은 약속, 계획은 실행"**

- 사양서는 **무엇**을 할 것인가 정의
- 구현 계획은 **어떻게** 할 것인가 구체화
- 작업은 **누가** **언제까지** 할 것인가 명시

**결과**:
- 사양서를 읽은 모든 사람이 같은 이해를 갖음
- 구현자는 명확한 로드맵을 가짐
- 관리자는 진행 상황을 투명하게 추적 가능
- 변경 영향을 신속하게 파악 가능

## ✅ 모범 사례

1. **항상 사양서 링크**: 계획과 작업이 원본을 참조하도록
2. **작은 단위로 분해**: 각 작업은 1-2일 내 완료 가능해야 함
3. **명확한 수용 기준**: 언제 "완료"인지 정의하기
4. **의존성 명시**: 선행 작업과 블로커 표시
5. **정기적 업데이트**: 활발한 작업 중에는 일일 진행 노트
6. **변경 추적**: 사양 변경과 그 영향 기록
7. **체크리스트 활용**: 시각적 진행도 표시
8. **결과물 링크**: PR, 설계, 문서를 작업에 연결
9. **단계별 마일스톤**: 명확한 중간 목표 설정
10. **진행 공유**: 팀에 정기적 진행 상황 공유

## ❓ FAQ

**Q: 사양서를 못 찾으면 어떻게 하나요?**

A: 다양한 방법 시도:
- 다양한 키워드로 재검색 (PRD, 스펙, 설계 등)
- 특정 teamspace로 범위 제한/확대
- 사용자에게 Notion URL 또는 ID 요청

**Q: 사양서가 여러 개면?**

A: 사용자에게 확인:
- 어떤 사양서를 구현할 것인지 물어보기
- 여러 옵션 제시
- 명확하지 않으면 각각에 대해 계획 수립

**Q: 작업 DB를 못 찾으면?**

A: 유연한 대응:
- "작업", "작업 관리", "Task Management" 등으로 검색
- 혹은 Slack, Jira 등 외부 도구 사용 제시
- 사용자에게 선호하는 위치 확인

**Q: 사양이 불명확하면?**

A: 명시하고 작업 생성:
- 계획에 "모호한 부분" 섹션 추가
- "명확화 필요" 작업 생성
- 사용자에게 더 많은 정보 요청

**Q: 요구사항이 충돌하면?**

A: 문서화하고 해결:
- 계획에 충돌사항 명시
- "의사결정 필요" 작업 생성
- 영향받는 작업 일시 중단 표시

**Q: 범위가 너무 크면?**

A: 단계적으로 분해:
- 큰 사양을 작은 사양들로 나누기
- 각 단계별 계획 수립
- 의존성 명시

**Q: 계획 중간에 사양이 변경되면?**

A: 신속하게 대응:
- 변경 사항 파악
- 영향받는 작업 식별
- 계획 업데이트
- 팀에 변경 공지
- 타임라인/우선순위 조정

**Q: 진행 상황은 얼마나 자주 업데이트해야 하나요?**

A: 상황에 따라:
- **활발한 작업**: 일일 진행 노트
- **주요 마일스톤**: 완료 시 계획 업데이트
- **상태 변경**: 작업 전환 시마다
- **주간 검토**: 관리자용 정기 요약

**Q: 여러 팀이 나뉘어 작업하면?**

A: 명확한 분할:
- 각 팀의 책임 영역 명시
- 팀 간 의존성 표시
- 교환점(interface) 명확하게 정의
- 주간 동기화 미팅 일정
